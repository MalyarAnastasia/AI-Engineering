# HW14 – эмбеддинги, FAISS, оценка retrieval и mini-RAG по базе знаний

## 1. Кратко: что сделано

- **База знаний:** синтетическая учебная подборка коротких статей в `data/knowledge_base.json` на одну тему — Git/GitHub, сдача заданий, тесты и CI в контексте учебного курса. Тема связная, по ней естественно формулировать вопросы и проверять поиск.
- **Чанкинг:** скользящее окно по символам с параметрами **chunk_size = 300**, **overlap = 50**; перекрытие уменьшает потерю контекста на границах фрагментов.
- **Векторизация:** модель **sentence-transformers/all-MiniLM-L6-v2**; векторы **L2-нормируются**, поиск — **косинусная близость** через **inner product** (`IndexFlatIP`).
- **Базовая проверка retrieval:** на контрольных запросах при **k = 5** получено **hit@5 = 0.70** и **recall@5 = 0.70**; дополнительно **MRR@5 ≈ 0.353**.
- **После обновления базы:** добавлены три документа про эмбеддинги, FAISS, метрики и переиндексацию (`data/knowledge_base_update.json`). Для запросов по семинару 14 и метрикам retrieval в top-5 появляются новые источники; см. `./artifacts/retrieval_before_after_update.csv`.
- **Mini-RAG:** извлечение top-k по FAISS, контекст из нескольких чанков, **извлекательный** ответ (предложение с максимальным лексическим пересечением с вопросом) и список источников; примеры в `./artifacts/rag_examples.csv`.

## 4. Retrieval и индекс

### 4.1. Эмбеддинги и индекс

- **Какая embedding-модель или способ векторизации использовались:** `sentence-transformers/all-MiniLM-L6-v2`.
- **Какая метрика сходства использовалась:** косинусная близость (векторы нормированы, скалярное произведение = косинус).
- **Какой индекс `FAISS` использовался:** `faiss.IndexFlatIP` (точный поиск по внутреннему произведению).
- **Какой `top_k` использовался по умолчанию:** **5**.
- **Что показали первые примеры retrieval:** для демо-запросов в top-k попадают статьи с близкой тематикой; конкретные источники и фрагменты выводятся в ноутбуке в секции 2.3.4.

### 4.2. Контрольные запросы и оценка retrieval

- **Сколько контрольных запросов использовалось:** **10**.
- **Что считалось релевантным ответом:** наличие чанка из **ожидаемого документа** (`doc_id`) среди top-**k** результатов.
- **Какие метрики retrieval считались:** **hit@k**, **recall@k** (при одном эталонном документе на запрос совпадают с долей успешных запросов), дополнительно **MRR@k**.
- **Итоговый `hit@k`:** **0.70** при **k = 5**.
- **Итоговый `recall@k`:** **0.70** при **k = 5**.
- **Дополнительные метрики (если были):** **MRR@5 ≈ 0.353**; сводка в `./artifacts/retrieval_metrics_summary.json`.
- **Какие запросы оказались самыми простыми:** запросы, где эталонный документ оказался на **1-й** позиции — например, про **gitignore** и про **сдачу заданий в homeworks** (см. `./artifacts/retrieval_eval.csv`).
- **Какие запросы оказались самыми проблемными:** запросы про **слияние ветки** и про **отправку коммитов на сервер** — эталонный источник не попал в top-5; также запрос про **правила оформления ноутбуков** — промах по top-5.

## 5. Эксперимент по параметрам retrieval и обновление базы знаний

- **Какой параметр сравнивался:** **`chunk_size`** при фиксированном **`overlap = 40`**.
- **Какие два варианта сравнивались:** **200** и **400** символов.
- **Как изменилось качество retrieval:** **hit@5** вырос с **0.600** (chunk_size=200) до **0.700** (chunk_size=400) на том же наборе контрольных запросов.
- **Какой вариант вы выбрали как основной и почему:** в основном пайплайне использованы **chunk_size = 300** и **overlap = 50** как разумный компромисс между гранулярностью и полнотой контекста; эксперимент показал, что слишком мелкие чанки (200) ухудшают hit@5 на этой базе.
- **Что именно было обновлено в базе знаний:** добавлены **три** документа про эмбеддинги/FAISS, метрики retrieval и переиндексацию после обновления корпуса.
- **Как была выполнена переиндексация:** полное пересоздание чанков по объединённому списку документов и повторное построение индекса FAISS.
- **Что изменилось в retrieval после обновления базы знаний:** для запросов про векторизацию, **hit@k**, **FAISS** и переиндексацию в top-5 появились новые файлы-источники (`kb/seminar14_embeddings_faiss.md`, `kb/rag_retrieval_metrics.md`, `kb/kb_update_reindex.md`); см. `./artifacts/retrieval_before_after_update.csv`.

## 7. Анализ

База знаний выбрана удачно для учебного цикла: она компактная, тематически цельная и при этом допускает содержательные вопросы. Чанкинг с перекрытием заметно влияет на то, окажется ли эталонный документ в одном фрагменте с ключевыми формулировками; эксперимент с **chunk_size** показал, что слишком короткие фрагменты снижают **hit@5**. Формальная оценка (**hit@k**, **recall@k**, **MRR@k**) полезна тем, что делает ошибки retrieval измеримыми и сопоставимыми между настройками. Обновление корпуса наглядно меняет выдачу по запросам, которых раньше не закрывала база (материалы семинара 14). Ошибки mini-RAG в основном следуют из retrieval: если в top попадают близкие по словам, но не по задаче предложения, ответ будет выглядеть правдоподобно, но фактически нерелевантным. Ограничения решения: плоский индекс без квантизации, одна лёгкая модель эмбеддингов, отсутствие reranking и генеративной постановки ответа на русском.

## 8. Итоговый вывод

В качестве базового контура для подобной учебной задачи разумно оставить **нормированные эмбеддинги + FAISS IndexFlatIP + фиксированный chunking с overlap** и измерять **hit@k** / **recall@k** на явном наборе запросов. Качество retrieval — главный «потолок» для mini-RAG: без попадания правильных фрагментов в top-k даже аккуратная сборка контекста не спасает. Связь retrieval и итогового ответа в нашем пайплайне прямая: извлекательный ответ почти не исправляет ошибки поиска, поэтому улучшение индекса и чанкинга важнее тонкой настройки текстовой шапки ответа.
